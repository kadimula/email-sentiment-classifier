Gmail → Sheets Agent

A multi-tenant Trigger.dev app: each client links their own Gmail and Google Sheets once (via Composio-hosted OAuth); a per-client cron then polls the inbox, keeps mail from a watched sender, and appends each match as a row to a target sheet. No web server, no database — all state lives in Trigger.dev + Composio.

✕

Google refresh tokens or client secrets — Composio holds and refreshes them. There is no googleapis dependency. (The GOOGLE_* keys in .env are dead leftovers; see below.)

Two independent grants under one userId: gmail (read) and googlesheets (write). Onboarding blocks until both are ACTIVE.

Dev vs. prod

The only environment split the code shows is the Trigger.dev secret-key prefix (tr_dev_ / tr_prod_), chosen by the --prod flag in the CLI scripts; that decides which Trigger.dev environment a run — and its schedules + env vars — lands in. WATCH_SENDER and TARGET_SHEET_ID are set per Trigger.dev environment from the dashboard. Composio userIds and their connections are not partitioned by Trigger environment — the same linked account is visible from both dev and prod runs, so choose your own userId convention if you want isolation. trigger dev runs the worker locally; trigger deploy ships it.
